# 主封单跟随策略触发参数详解

本文档只解释 `MainSealFollowStrategy` 的入场触发参数。

核心结论：

```text
满足基础前提
and _main_seal_ok()
=> 触发两层下单
```

对应代码：

- `strategy/main_seal_follow_strategy.py::_maybe_trigger_entry`
- `strategy/main_seal_follow_strategy.py::_main_seal_ok`

## 一、触发链路的顺序

策略不是“看到封单就排”，而是先后过三层过滤。

第一层：基础前提

- 当前状态必须是 `WAIT_SIGNAL`
- 不能已有持仓
- 不能已有活动挂单
- 已经算出涨停价
- 当前必须有一份有效的买一队列快照
- 不能处在撤单冷却期

第二层：`main_seal` 条件

- 确认涨停价买一队列前排确实有大单
- 确认最近有新的涨停价大买单补进来
- 确认最近主力最新动作不是大撤买

只有两层都通过，才会触发 `probe + main` 两层下单。

## 三、当前版本不再使用的扫板参数

为了先把“排板”逻辑做稳定，当前版本的入场触发不再依赖 `sweep`。

也就是说，下面这些参数目前保留在配置里，但**不再是入场前置条件**：

- `sweep_window_ms`
- `sweep_near_limit_ticks`
- `sweep_min_amount`

当前实际触发语义是：

```text
买一队列价格已经是涨停价
and 主封结构条件成立
=> 触发两层下单
```

因此，现阶段是“只做排板，不做扫板”。

下面保留这些参数的解释，仅作为后续可能恢复扫板逻辑时的参考。

### 1. `sweep_window_ms`

含义：

- 扫描最近成交的时间窗口，单位毫秒。
- 当前默认值是 `1200`，即最近 `1.2` 秒。

它控制什么：

- 窗口越大，越容易把分散的成交累加进去，触发更宽松。
- 窗口越小，越强调“瞬时冲板”。

实际代码逻辑：

- 取最近 `sweep_window_ms` 毫秒内的成交记录。
- 这些成交来自 `_recent_trades` 缓存。

例子：

- 如果 `sweep_window_ms=1200`，那就只看最近 `1.2` 秒。
- `1.2` 秒之前的成交，无论金额多大，都不参与这次触发判断。

### 2. `sweep_near_limit_ticks`

含义：

- 只统计离涨停价足够近的成交。
- 当前默认值是 `3`，表示只看涨停价下方 `3` 个 tick 以内。

A 股默认 tick：

- `1 tick = 0.01`

换算方式：

- 如果涨停价是 `10.00`
- 且 `sweep_near_limit_ticks=3`
- 那么只看 `9.97` 到 `10.00` 之间的成交

它控制什么：

- 数值越小，要求成交更贴近涨停价，触发更严格。
- 数值越大，会把更下面的成交也统计进来，触发更宽松。

### 3. `sweep_min_amount`

含义：

- 在 `sweep_window_ms` 这个窗口内、在接近涨停价的价格区间里，累计成交金额至少要达到多少。

当前默认值：

- `5000000`，即 `500 万`

它控制什么：

- 用来过滤“轻飘飘摸板”
- 只接受“短时间内确实有足够多成交资金往上打”的情况

例子一：通过

假设涨停价是 `10.00`，最近 `1.2` 秒成交如下：

- `9.98` 成交 `150 万`
- `9.99` 成交 `180 万`
- `10.00` 成交 `220 万`

因为：

- 三笔都在 `9.97` 到 `10.00` 区间内
- 总金额 `550 万`
- 并且确实有一笔成交打到 `10.00`

所以 `sweep` 通过。

例子二：不通过

- `9.98` 成交 `100 万`
- `9.99` 成交 `120 万`
- `10.00` 成交 `80 万`

虽然也打到了涨停价，但总金额只有 `300 万`，低于 `500 万`，所以不通过。

### 4. `main_seal_window_ms`

含义：

- “最近窗口”的长度，单位毫秒。
- 当前默认值是 `1000`，即最近 `1` 秒。

它主要给两类判断使用：

- 最近是否出现过涨停价大买单
- 最近是否出现过大撤买，且撤买发生在最近那笔大买单之后

它不是直接判断队列形状，而是判断最近这段时间里，主封资金有没有继续动作。

### 5. `big_amount_min`

含义：

- 统一的大单金额门槛。
- 当前默认值是 `2000000`，即 `200 万`

这个参数同时影响三件事：

1. 队列里多大才算“大封单”
2. `l2order` 里多大才算“涨停价大买单”
3. `l2transaction + l2order` 回查后，多大才算“涨停价大撤买”

例子：

- 涨停价是 `10.00`
- `big_amount_min=2000000`

那么单笔委托至少需要：

```text
2,000,000 / 10.00 = 200,000 股 = 2000 手
```

也就是说：

- 买一队列里单笔 `2000 手` 以上，才算“大单”

调参含义：

- 设太低：很多普通挂单都会被算成大单，主封识别会失真
- 设太高：小票里可能根本看不到大单，几乎不会触发

### 6. `require_recent_big_limit_buy`

含义：

- 是否要求最近 `main_seal_window_ms` 窗口内，必须出现过涨停价大买单。

当前默认值：

- `true`

为什么需要它：

- 只看当前队列厚不厚还不够
- 有些票看起来买一很厚，但只是老单挂在那里，最近没有新的封单资金继续补进来

当前逻辑：

- 如果打开这个参数
- 最近窗口里没有记录到涨停价大买单
- 那么 `_main_seal_ok()` 直接失败

例子：

- 当前买一前排确实有大单
- 但最近 `1` 秒内没有新的涨停价大买单

则：

- 队列形状像主封
- 但仍然不触发

这条的本质是：

- 不仅要“盘面像封单”
- 还要“最近真的有人继续往涨停价补钱”

### 7. `block_on_recent_big_limit_cancel`

含义：

- 如果最近窗口内出现了“大撤买”压过最近的大买单动作，就阻断触发。

当前默认值：

- `true`

当前代码逻辑：

1. 取最近 `main_seal_window_ms` 内的大撤买
2. 取最近 `main_seal_window_ms` 内的大买单
3. 如果没有大买单，但有大撤买，直接阻断
4. 如果有大买单，但某笔大撤买发生在最近那笔大买单之后，也阻断

这条不是单纯地看“有没有撤单”，而是在看：

```text
最近主力最后一个大动作，到底是补封还是撤封
```

例子一：阻断

- `14:01:00.100` 有一笔涨停价大买单 `300 万`
- `14:01:00.600` 有一笔涨停价大撤买 `260 万`

最新动作是撤买，所以阻断触发。

例子二：不阻断

- `14:01:00.100` 有一笔大撤买 `260 万`
- `14:01:00.700` 有一笔大买单 `400 万`

最新动作是补单，所以不阻断。

### 8. `main_seal_front_max_index`

含义：

- 要求买一前 50 笔可见队列中，第一笔大单必须足够靠前。

当前默认值：

- `5`

当前代码的判定方式：

- 找出前 50 笔队列里所有达到 `big_amount_min` 的挂单
- 取第一笔大单的位置
- 只有当该位置 `<= main_seal_front_max_index` 才通过

这里有一个必须明确的实现细节：

- 当前代码用的是 `0` 基索引

也就是说：

- `0` 表示第 `1` 笔
- `1` 表示第 `2` 笔
- `5` 表示第 `6` 笔

因此，当前配置 `main_seal_front_max_index=5` 的实际含义是：

```text
第一笔大单出现在前 6 笔内，就算通过
```

不是严格的“前 5 笔”。

例子一：通过

- 第1笔 `300 手`
- 第2笔 `500 手`
- 第3笔 `2200 手`

如果涨停价换算后 `2200 手` 已达到 `big_amount_min`

则：

- 第一笔大单位置索引是 `2`
- `2 <= 5`
- 通过

例子二：边界也通过

- 第6笔才出现第一笔大单

则：

- 它的索引是 `5`
- 按当前代码仍然通过

例子三：不通过

- 第7笔才出现第一笔大单

则：

- 索引是 `6`
- `6 > 5`
- 不通过

这条的本质是：

- 主封单要顶在前排
- 不能只是前 50 笔深处藏着一笔大单

### 9. 已涨停启动补充分支

交易时间实测发现：如果程序启动时股票已经封住涨停，`l2orderqueue` 可以看到当前前 50 笔队列和总排队笔数，但 `l2order` 不一定在 `main_seal_window_ms` 内继续出现新的 `big_amount_min` 以上涨停价大买单。

这个场景和当前目标不一致：策略要做的是“板后尽量靠前排队”，不是在已经封住很久、队列很深之后再去排。因此当前正式参数保持 `require_recent_big_limit_buy=true`，不会仅凭既有队列触发。

代码里保留了一个显式实验分支，但默认关闭：

```text
买一价格 = 涨停价
and orderqueue 价格 = 涨停价
and 前 50 队列里第一笔大单足够靠前
and 持续观察 existing_limit_observe_ms 后仍然稳定
and 最近没有大撤买阻断
=> 可以认为主封结构成立
```

相关参数：

- `allow_existing_limit_queue=false`：是否启用“启动时已涨停”的队列补充分支。正式运行不建议打开。
- `existing_limit_observe_ms=3000`：已涨停队列至少连续稳定观察多久，默认 3 秒。设为 0 只适合测试。
- `existing_limit_min_bid1_lot=0`：买一总量下限，单位是手；0 表示不额外限制。
- `existing_limit_min_reported_orders=0`：`l2orderqueue.bidLevelNumber` 的总排队笔数下限；0 表示不额外限制。

当前验收标准：启动时已经涨停很久、没有最近涨停价大买单补进来的股票，不触发排板。

### 10. 观察单未成交处理

当前简化处理：观察单不成交就一直等，不因为 `probe_wait_timeout_ms` 独立撤单。

相关参数：

- `cancel_before_probe_fill_on_timeout=false`：观察单超时前是否允许按市场指标撤单。默认关闭。
- `probe_wait_timeout_ms=8000`：保留为未来启用超时撤单时的观察窗口；当前默认不触发撤单。

这样做的原因是：观察单是否成交本身就是关键观测点。没有成交说明还没轮到我们，不应该用一个固定秒数强制撤。

### 11. `back_big_min_amount`

这个参数需要特别说明。

从设计含义上看，它像是：

- 在前排大单之后，后面是否还有足够的承接大单

但是按当前代码实际状态，它并不在新的主触发链路里。

也就是说，当前真正决定入场触发的这条链路：

- `_maybe_trigger_entry()`
- `_is_sweep_ok()`
- `_main_seal_ok()`

里面没有直接使用 `back_big_min_amount`。

它目前主要还用于旧的队列位置分析逻辑 `_analyze_queue_and_maybe_cancel()`。

所以如果你现在是在调“是否触发入场”，这个参数不是第一优先级。

结论：

- 文档含义上，它属于封单结构判断参数
- 但当前代码里，它还没有真正接进新的触发核心

## 四、完整触发例子

假设：

- 涨停价 `10.00`
- `big_amount_min=200 万`
- `main_seal_window_ms=1000`
- `main_seal_front_max_index=5`

当前买一前几笔队列：

- 第1笔 `300 手`
- 第2笔 `500 手`
- 第3笔 `2400 手`

如果涨停价是 `10.00`，那么：

- `2400 手 = 240,000 股`
- 对应金额 `240 万`
- 超过 `big_amount_min=200 万`

所以：

- 第一笔大单出现在第 3 笔
- 索引是 `2`
- 满足 `2 <= 5`

队列前排大单条件通过。

最近 `1` 秒 `l2order`：

- 有一笔涨停价新增买单 `250 万`

因此：

- `require_recent_big_limit_buy=true` 通过

最近 `1` 秒 `l2transaction` 回查后：

- 没有发现晚于这笔大买单的大撤买

因此：

- `block_on_recent_big_limit_cancel=true` 通过

最终：

- `main_seal` 通过
- 触发 `probe + main` 两层下单

## 五、如何理解当前触发逻辑

可以把当前版本理解成两层过滤器。

第一层：

- `big_amount_min`
- `main_seal_front_max_index`

这层回答的问题是：

```text
当前封单前排是不是有真大单顶着？
```

第二层：

- `main_seal_window_ms`
- `require_recent_big_limit_buy`
- `block_on_recent_big_limit_cancel`

这层回答的问题是：

```text
最近主力动作是继续补封，还是已经开始撤封？
```

只有两层答案都偏正面，策略才会进场排板。
## 六、Level2 订阅分层

为避免股票池扩大后完整 Level2 订阅过重，当前采用两层订阅：

- 全股票池默认只订阅 `l2quote`，用于轻量扫描最新价、买一价和涨停价。
- 当 `last_price` 或 `bid1` 进入 `quote_trigger_near_limit_ticks` 定义的接近涨停区间后，才动态打开 `l2order`、`l2transaction`、`l2orderqueue`。
- 打开详细 Level2 后，本轮运行中保持订阅，不做频繁退订，避免订阅抖动。

触发判断仍然不变：完整 Level2 打开后，还必须看到涨停价新增大买单、队列前排大单和撤单阻断条件都通过，才会真正排板。
