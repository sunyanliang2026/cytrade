# 主封跟随策略设计方案

本文档定义 `MainSealFollowStrategy` 的实现方案。设计依据见
`docs/LEVEL2_VALIDATION.md`。

## 设计目标

策略目标是在涨停封单已经形成后，跟随主封资金排队买入。当前 QMT Level2
只能稳定看到买一前 50 笔队列，不能还原完整排队位置，因此策略不再依赖完整
队列推断，而是用“观测单成交”作为主单进入临近成交区的关键时点。

核心判断：

- 前 50 笔队列只作为可见窗口，不当作完整队列。
- 观测单和主单连续提交，形成自己的排队片段。
- 观测单成交说明队列已经推进到自己的订单区域。
- 观测单成交后，立即根据市场强弱判断主单保留还是撤单。
- 不设计后续追加主单。下单只有两层：观测单和主单。

## 下单结构

下单分两层：

| 层级 | 名称 | 作用 | 数量 |
|---|---|---|---|
| 第一层 | 观测单 | 用成交反馈确认队列推进到自己的订单区域 | 通常 1 到 2 手 |
| 第二层 | 主单 | 实际计划买入的主体仓位 | 计划仓位扣除观测单后的剩余部分 |

下单顺序：

1. 策略信号成立。
2. 先提交观测单。
3. 紧接着提交主单。
4. 进入等待观测单成交状态。
5. 观测单成交后进入主单决策窗口。
6. 根据市场强弱决定保留主单或撤主单。

注意：

- 观测单成交不是“再下主单”的触发点。
- 主单必须已经在观测单后面排队。
- 没有第三层，也没有后续追加主单。
- “封板打开再撤”不作为独立规则。开板如果导致订单成交，按成交结果处理；
  是否撤剩余主单由观测单成交时点附近的市场强弱规则决定。

## 状态机

| 状态 | 中文含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| `WAIT_SIGNAL` | 等待信号 | 初始状态或上一轮结束 | 主封信号成立 |
| `CHAIN_SUBMITTED` | 两层订单已提交 | 观测单和主单提交成功 | 收到订单回报 |
| `WAIT_PROBE_FILL` | 等待观测单成交 | 观测单已报入，主单已报入 | 观测单成交、超时、主单异常成交或撤单触发 |
| `PROBE_FILLED_DECISION` | 观测单成交后的主单决策 | 观测单发生成交 | 主单保留或主单撤单 |
| `MAIN_KEEPING` | 主单保留等待成交 | 决策窗口判断市场仍强 | 主单成交、部分成交后撤剩余、或市场转弱撤单 |
| `MAIN_CANCELING` | 主单撤单中 | 决策窗口判断市场转弱 | 撤单回报完成 |
| `HAS_POSITION` | 已有持仓 | 观测单或主单产生有效成交 | 策略退出或持仓处理完成 |
| `EXITED` | 策略结束 | 已退出或不再参与 | 无 |

## Level2 数据规则

### l2order

`l2order` 用于识别新增委托。

| 字段 | 中文含义 | 使用规则 |
|---|---|---|
| `price` | 委托价格 | 等于涨停价时，计入涨停价委托 |
| `volume` | 委托数量，单位股 | 用于计算金额 |
| `entrustNo` | 委托编号 | 用于和撤单记录关联 |
| `entrustType` | 委托类型 | `1` 表示正常委托 |
| `entrustDirection` | 委托方向 | `1` 买入，`2` 卖出 |

涨停价新增买单：

```text
l2order.price == limit_up_price
and l2order.entrustDirection == 1
```

涨停价新增卖单：

```text
l2order.price == limit_up_price
and l2order.entrustDirection == 2
```

### l2transaction

`l2transaction` 用于识别成交和深市撤单。

| 字段 | 中文含义 | 使用规则 |
|---|---|---|
| `price` | 成交价格 | 成交记录可用；深市撤单记录通常为 `0.0` |
| `volume` | 成交或撤单数量，单位股 | 撤单金额要结合原始委托价格计算 |
| `amount` | 成交金额 | 深市撤单记录通常为 `0.0` |
| `buyNo` | 买方委托编号 | 深市撤买时用于反查原始买单 |
| `sellNo` | 卖方委托编号 | 深市撤卖时用于反查原始卖单 |
| `tradeFlag` | 成交标志 | 深市 `3` 表示撤单 |

深市撤单识别：

```text
l2transaction.tradeFlag == 3
```

深市涨停价撤买确认：

```text
l2transaction.tradeFlag == 3
and l2transaction.buyNo maps to l2order.entrustNo
and matched_l2order.price == limit_up_price
and matched_l2order.entrustDirection == 1
```

撤单金额：

```text
matched_l2order.price * l2transaction.volume
```

如果 `buyNo` 或 `sellNo` 找不到原始委托，记为 `UNKNOWN_CANCEL`。未知撤单只作为风险
信号，不计入确认的涨停价撤买金额。

### l2orderqueue

`l2orderqueue` 只作为买一前 50 笔可见窗口。

| 字段 | 中文含义 | 使用规则 |
|---|---|---|
| `bidLevelPrice` | 买一队列价格 | 等于涨停价时才用于封单判断 |
| `bidLevelVolume` | 买一可见队列数量，单位手 | 当前可见前 50 笔 |
| `bidLevelNumber` | 买一总委托笔数 | 用于判断队列是否只看到部分 |

派生字段：

| 字段 | 中文含义 | 计算方式 |
|---|---|---|
| `observed_queue_count` | 可见委托笔数 | `len(bidLevelVolume)` |
| `reported_total_order_count` | 源数据报告的买一总委托笔数 | `bidLevelNumber` |
| `is_partial_queue` | 是否部分队列 | `bidLevelNumber > len(bidLevelVolume)` |
| `front50_depth_lot` | 前 50 笔可见总量，单位手 | `sum(bidLevelVolume)` |

## 涨停场景下三类 Level2 数据的联动理解

在“买一就是涨停价”的排板场景里，`l2order`、`l2transaction`、`l2orderqueue`
分别表达的是不同层面的信息。

### 总表

| 数据 | 它表示什么 | 涨停封板时最常见含义 | 对策略的作用 | 你该怎么读 |
|---|---|---|---|---|
| `l2order` | 新增委托 | 新的买单继续往涨停价买一队列后面排；新的卖单会形成抛压 | 看补封、看新增抛压 | 它看的是“又来了什么新单” |
| `l2transaction` | 已发生的成交或撤单 | 正常成交通常表示卖盘被封单承接；`tradeFlag=3` 表示深市撤单 | 看封单被消耗速度、看撤买 | 它看的是“刚刚实际发生了什么结果” |
| `l2orderqueue` | 买一前 50 笔可见队列 | 当前封单前排窗口，不是完整队列 | 看前排封单结构强弱 | 它看的是“此刻队列长什么样” |

### 分场景理解

| 场景 | `l2order` | `l2transaction` | `l2orderqueue` | 盘口含义 |
|---|---|---|---|---|
| 新的涨停价买单来了 | 会看到 `price=涨停价` 且 `entrustDirection=1` | 不一定立刻有变化 | 买一后排继续增厚 | 有人继续排板、补封 |
| 新的涨停价卖单来了 | 会看到 `price=涨停价` 且 `entrustDirection=2` | 如果立刻被买一吃掉，就会很快看到成交 | 前排队列可能被消耗 | 有抛压出来，封单在承接 |
| 涨停价不断成交 | 不一定每次都有新 `l2order` | 会持续出现涨停价成交 | 前排可见队列会逐步变薄或重排 | 封单正在被卖盘消耗 |
| 深市大撤买 | 原始买单早先在 `l2order` 里 | `tradeFlag=3`，靠 `buyNo/sellNo` 回查原始委托 | 队列可能同步走弱 | 主封资金在撤退 |
| 队列看起来还很厚 | 不代表刚有新买单 | 不代表刚有新成交 | 只说明“当前前 50 笔窗口还厚” | 静态强，不等于动态强 |

### 口诀

- `l2order`：看“新单”
- `l2transaction`：看“结果”
- `l2orderqueue`：看“当前排队截面”

### 放到本策略里的使用口径

| 指标目的 | 主要看哪类数据 |
|---|---|
| 有没有继续补封 | `l2order` 的涨停价新增买单 |
| 有没有明显撤封 | `l2transaction` 的深市撤买 |
| 封单是不是在被快速消耗 | `l2transaction` 的涨停价成交金额 |
| 当前前排封单还强不强 | `l2orderqueue` 的前 50 笔结构 |

## 指标定义

所有窗口指标默认在“当前观察窗口”内计算。观察窗口包括两类：

- 观测单等待窗口：从两层订单提交后开始，到观测单成交或超时。
- 观测单成交决策窗口：从观测单成交后开始，持续很短时间，用于决定主单留撤。

| 指标 | 中文解释 | 来源 | 用途 |
|---|---|---|---|
| `probe_fill_ms` | 观测单从报入到首次成交的耗时，单位毫秒 | 订单回报 | 判断自己的订单区域是否已经接近成交 |
| `limit_buy_add_amount` | 涨停价新增买单金额 | `l2order` | 判断封单是否继续补强 |
| `limit_sell_add_amount` | 涨停价新增卖单金额 | `l2order` | 判断涨停价抛压 |
| `limit_buy_cancel_amount` | 已确认的涨停价买单撤单金额 | `l2transaction + l2order索引` | 判断封单资金是否撤退 |
| `limit_buy_net_amount` | 涨停价买单净增强金额 | 新增买单金额减撤买金额 | 判断主封是否仍在增强 |
| `cancel_to_add_ratio` | 撤买金额和新增买单金额的比例 | 撤买金额 / 新增买单金额 | 判断撤单压力是否过大 |
| `limit_trade_amount` | 涨停价成交金额 | `l2transaction` | 判断封单被卖单消耗的速度 |
| `unknown_cancel_amount` | 未能反查原始委托的撤单数量折算风险量 | `tradeFlag=3` 未匹配记录 | 判断未知撤单风险 |
| `bid1_volume_delta_lot` | 买一总挂单量变化，单位手 | `l2quote.bidVol[0]` | 判断封单总量是否变薄 |
| `bid_level_number_delta` | 买一总委托笔数变化 | `l2orderqueue.bidLevelNumber` | 判断排队单数是否塌陷 |
| `front50_depth_delta_lot` | 前 50 笔可见队列总量变化，单位手 | `l2orderqueue.bidLevelVolume` | 判断可见队列是否变弱 |

## 参数定义

| 参数 | 中文解释 | 默认建议 | 备注 |
|---|---|---:|---|
| `probe_lots` | 观测单手数 | `1` | 用于探测成交临界点，不承担主要仓位 |
| `main_lots` | 主单手数 | 按计划金额计算 | 主单为实际买入主体 |
| `submit_gap_ms` | 观测单和主单提交间隔，单位毫秒 | `0` 到 `200` | 越短越接近连续排队 |
| `probe_wait_timeout_ms` | 等待观测单成交的最长时间 | `8000` | 超时说明主单不在临近成交区 |
| `probe_decision_window_ms` | 观测单成交后的决策窗口 | `800` 到 `1500` | 在这个窗口内判断主单留撤 |
| `post_probe_keep_ms` | 观测单成交后主单保留观察时间 | `3000` 到 `8000` | 主单保留后继续监控市场强弱 |
| `big_amount_min` | 大单金额门槛 | 可配置 | 不同股票差异大，不固定为 200 万 |
| `cancel_to_add_ratio_max` | 撤买金额占新增买单金额的最大允许比例 | `0.6` | 超过说明撤单压力偏大 |
| `bid1_volume_drop_ratio_max` | 买一总量最大允许下降比例 | `0.35` | 用于判断封单总量是否明显变薄 |
| `front50_depth_drop_ratio_max` | 前 50 笔可见队列最大允许下降比例 | `0.35` | 用于判断可见窗口是否明显变弱 |
| `unknown_cancel_risk_amount` | 未知撤单风险金额门槛 | 可配置 | 未匹配撤单过大时提高风险等级 |
| `min_limit_buy_net_amount` | 最小涨停价买单净增强金额 | 可配置 | 低于该值说明补单不足 |

## 主单留撤规则

### 观测单未成交前

观测单未成交前，策略不认为主单已经进入临近成交区。

撤主单条件：

- 等待超过 `probe_wait_timeout_ms`。
- 涨停价确认撤买金额持续大于新增买单金额。
- 买一总量和前 50 笔可见队列同步明显变弱。
- 未知撤单金额过大，且盘口/队列指标同步变弱。

### 观测单成交后

观测单成交后，说明队列已经推进到自己的订单区域，立即进入
`PROBE_FILLED_DECISION`。

保留主单条件：

- `limit_buy_net_amount` 为正，或者撤买金额没有明显压过新增买单金额。
- `cancel_to_add_ratio <= cancel_to_add_ratio_max`。
- 买一总量没有超过允许幅度下滑。
- 前 50 笔可见队列没有超过允许幅度下滑。
- 未知撤单金额没有达到风险门槛。

撤主单条件：

- 确认撤买金额明显大于新增买单金额。
- `cancel_to_add_ratio` 超过阈值。
- 涨停价成交消耗很大，但新增买单补不上。
- 买一总量、买一委托笔数、前 50 笔可见队列同步走弱。
- 未知撤单金额很大，且同时出现盘口或队列走弱。

主单发生部分成交时：

- 已成交部分进入持仓。
- 未成交剩余部分继续按上述规则决定保留或撤单。
- 不追加新主单。

## 实现顺序

1. 修正 Level2 数据解析。
   验收：`l2order.entrustDirection=1/2` 正常映射为 `BUY/SELL`。

2. 扩展 `l2transaction` 字段。
   验收：策略能读取 `tradeFlag`、`buyNo`、`sellNo`、`tradeIndex`。

3. 维护滚动委托号索引。
   验收：`tradeFlag=3 + buyNo` 能反查原始 `l2order`。

4. 生成派生撤单事件。
   验收：深市涨停价撤买能生成 `CANCEL_BUY`，价格来自原始委托，不是 `0.0`。

5. 扩展部分队列模型。
   验收：日志中能看到 `observed_queue_count`、`reported_total_order_count`、
   `is_partial_queue`。

6. 实现两层订单状态机。
   验收：策略能区分观测单和主单，观测单成交后进入主单决策窗口。

7. 实现主单留撤决策。
   验收：每次保留或撤单都输出指标快照和中文原因。

## 日志要求

每次关键动作必须记录中文解释：

- 提交观测单：记录手数、价格、订单号。
- 提交主单：记录手数、价格、订单号。
- 观测单成交：记录成交耗时、成交数量。
- 主单保留：记录保留原因和核心指标。
- 主单撤单：记录撤单原因和核心指标。
- 未知撤单：记录 `buyNo/sellNo`、数量、是否匹配失败。

主单决策日志至少包含：

- 涨停价新增买单金额。
- 涨停价撤买金额。
- 撤买/新增买比例。
- 涨停价成交金额。
- 买一总量变化。
- 买一总委托笔数变化。
- 前 50 笔可见队列变化。
- 未知撤单金额。
